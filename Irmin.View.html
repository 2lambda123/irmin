Module type Irmin.VIEW

module type VIEW = sig .. end
View provides an in-memory partial mirror of the store, with lazy reads and delayed write.

Views are like staging area in Git: they are temporary non-persistent areas (they disappear if the host crash), hold in memory for efficiency, where reads are done lazily and writes are done only when needed on commit: if if you modify a key twice, only the last change will be written to the store when you commit. Views also hold a list of operations, which are checked for conflicts on commits and are used to replay/rebase the view if needed. The most important feature of views is that they keep track of reads: i.e. you can have a conflict if a view reads a key which has been modified concurrently by someone else.

val empty : unit -> Irmin.t Lwt.t
Create an empty view. Empty views do not have associated backend configuration values, as they can perform in-memory operation, independently of any given backend.
val rebase : Irmin.t -> into:Irmin.t -> unit Irmin.Merge.result Lwt.t
rebase x t i rebases the actions done on the view t x into the view i x. If a read operation doesn't return the same result, return Conflict. Only the view i is updated.
val rebase_exn : Irmin.t -> into:Irmin.t -> unit Lwt.t
Same as Irmin.VIEW.rebase but raise Irmin.Merge.Conflict in case of conflict.
val of_path : db -> key -> Irmin.t Lwt.t
of_path t p reads the view from a path p in the branch t. This is a cheap operation, all the real reads operation will be done on-demand when the view is used.
val update_path : db -> key -> Irmin.t -> unit Lwt.t
update_path t p v replaces the sub-tree under p in the branch t by the contents of the view v. See Irmin.VIEW.merge_path for more details.
val rebase_path : db -> key -> Irmin.t -> unit Irmin.Merge.result Lwt.t
rebase_path t p v rebases the view v on top of the contents of the sub-tree under p in the branch t. Rebasing means re-applying every actions stored in v, including the reads. Return Irmin.Merge.Conflict if one of the action cannot apply cleanly. See Irmin.VIEW.merge_path for more details.
val rebase_path_exn : db -> key -> Irmin.t -> unit Lwt.t
Same as Irmin.VIEW.rebase_path but raise Irmin.Merge.Conflict in case of conflict.
val merge_path : db ->
       ?max_depth:int -> ?n:int -> key -> Irmin.t -> unit Irmin.Merge.result Lwt.t
merge_path t path v merges the view v with the contents of the sub-tree under p in the branch t. Merging means applying the merge function for map between the view's contents and t's sub-tree.

  • Irmin.VIEW.update_path discards any preexisting sub-tree.
  • Irmin.VIEW.rebase_path is operation based. It keeps track of read operations which can lead to conflicts. It replays the full operation history of the view on top of any preexisting sub-tree.
  • Irmin.VIEW.merge_path is state based. Is is an efficient 3-way merge operators between prefix trees, based on Irmin.Merge.Map.merge.

val merge_path_exn : db -> ?max_depth:int -> ?n:int -> key -> Irmin.t -> unit Lwt.t
Same as Irmin.VIEW.merge_path but raise Irmin.Merge.Conflict in case of conflicts.

Heads


type head
The type for commit heads.
val parents : Irmin.t -> head list
parents t are t's parent commits.
val make_head : db ->
       Irmin.task ->
       parents:head list -> contents:Irmin.t -> head Lwt.t
make_head t task ~parents ~contents creates a new commit into the store where the branch t is stored and return its id (of type Irmin.VIEW.head). The new commit has task as task and the given parents and contents. The actual parents of contents are not used.
val watch_path : db ->
       key ->
       ?init:head * Irmin.t ->
       ((head * Irmin.t) Irmin.diff -> unit Lwt.t) ->
       (unit -> unit Lwt.t) Lwt.t
watch_head t p f calls f every time subpaths of p are updated in the branch t. The callback parameters contains branch's current head and the corresponding view.
